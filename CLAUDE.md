# StockTrading 프로젝트 — Claude Code 규약

## 프로젝트 개요
주식 거래 애플리케이션입니다. 이 파일은 Claude Code가 이 저장소에서 작업할 때 따라야 할 규약과 규칙을 정의합니다.

---

## 작업 시작 절차
- 작업을 시작하기 전에 **반드시** `git pull`을 실행해 최신 상태를 가져온다
- 충돌이 발생하면 사용자에게 알리고 해결 방법을 확인한다

---

## 일반 원칙
- 깔끔하고 읽기 쉬우며 유지보수하기 쉬운 코드를 작성한다
- 영리함보다 단순함을 선호한다
- 명시적으로 요청된 것 이상의 기능이나 추상화를 추가하지 않는다
- 로직이 명확하지 않은 경우에만 주석을 추가한다
- 보안 취약점(SQL 인젝션, XSS, 커맨드 인젝션 등)을 절대 도입하지 않는다

---

## 코드 스타일

### 네이밍
- 변수 및 함수: `camelCase`
- 클래스 및 타입: `PascalCase`
- 상수: `UPPER_SNAKE_CASE`
- 파일: `kebab-case` (예: `order-service.ts`)
- 데이터베이스 컬럼: `snake_case`

### 구조
- 파일/모듈당 하나의 책임만 갖는다
- 함수는 짧고 집중적으로 유지한다 (50줄 이하를 목표로)
- 깊은 중첩을 피하고 얼리 리턴을 선호한다
- 관련 파일은 타입이 아닌 기능별로 그룹화한다

### 에러 처리
- 항상 명시적으로 에러를 처리한다 — 조용히 무시하지 않는다
- 충분한 컨텍스트(무엇이, 어디서, 어떤 입력으로 실패했는지)와 함께 에러를 로깅한다
- 호출자에게 의미 있는 에러 메시지를 반환한다

## Git 규약

### 브랜치 네이밍
- `feat/<짧은-설명>` — 새로운 기능
- `fix/<짧은-설명>` — 버그 수정
- `refactor/<짧은-설명>` — 리팩토링
- `chore/<짧은-설명>` — 툴링, 설정, 의존성

### 커밋 메시지
Conventional Commits 형식을 따른다:
```
<타입>(<범위>): <짧은 요약>

[선택적 본문]
```
타입: `feat`, `fix`, `refactor`, `chore`, `docs`, `test`

예시:
```
feat(order): 시장가 주문 실행 추가
fix(portfolio): 미실현 손익 계산 수정
chore(deps): 거래 API 클라이언트 v3로 업그레이드
```

### Pull Requests
- PR은 작고 단일 관심사에 집중되게 유지한다
- 무엇이 변경되었고 왜 변경되었는지 짧은 요약을 포함한다
- 병합 전에 모든 CI 검사가 통과해야 한다

---

## 보안 규칙
- API 키, 시크릿, 자격증명을 **절대** 커밋하지 않는다 — 환경 변수를 사용한다
- 시스템 경계에서 항상 사용자 입력을 검증하고 정제한다
- 모든 데이터베이스 작업에 파라미터화된 쿼리를 사용한다
- 민감한 설정은 `.env` 파일에 보관한다 (커밋하지 않는다)
- 처음부터 `.env`를 `.gitignore`에 추가한다

---

## 테스팅
- 비즈니스 로직, 특히 금융 계산에 대한 테스트를 작성한다
- 단위 테스트는 실제 외부 API나 데이터베이스를 호출하지 않는다 — 목/스텁을 사용한다
- 통합 테스트는 전용 테스트 데이터베이스를 사용할 수 있다
- 테스트 파일명: 언어에 따라 `<모듈>.test.ts` 또는 `<모듈>_test.go`

---

## 도메인 특화 규칙 (주식 거래)
- 반올림 오류를 피하기 위해 모든 금액은 정밀한 소수 타입을 사용한다 (부동소수점 절대 금지)
- 주문 흐름: 검증 → 실행 → 기록 (단계를 절대 건너뛰지 않는다)
- 금융 계산(손익, 수익률, 수수료)은 반드시 단위 테스트해야 한다
- 시장 데이터 피드는 잠재적으로 신뢰할 수 없는 것으로 취급한다 — 데이터 누락과 오류를 우아하게 처리한다
- 모든 거래 실행에 대한 감사 로그는 필수이다

---

## 환경 설정
- 실행 전에 `.env.example`을 `.env`로 복사하고 필요한 값을 입력한다
- `.env.example`에 모든 필수 환경 변수를 문서화한다
- 해당되는 경우 로컬 개발 의존성(DB, 캐시 등)에 Docker Compose를 사용한다

---

## Claude가 하지 말아야 할 것
- 확인 없이 `main`/`master`에 직접 푸시하지 않는다
- 명시적인 사용자 승인 없이 데이터베이스 테이블을 삭제하거나 비우지 않는다
- 확인 없이 파괴적인 명령(`rm -rf`, `git reset --hard` 등)을 실행하지 않는다
- 민감한 데이터(토큰, 비밀번호, 개인정보)를 로깅하지 않는다
